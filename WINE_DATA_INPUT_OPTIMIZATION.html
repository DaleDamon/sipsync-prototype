Wine Data Input Optimization

SipSync Internal Brainstorming Document • February 2026

The Problem: Manually inputting Quartino's 53 wines into CSV format required filling in 11 fields per wine — roughly 580 manual data entries for a single restaurant. The hardest part is the sensory profile (acidity, tannins, body, sweetness, flavor notes), which requires wine knowledge and can't be found on a typical restaurant menu.

For SipSync to scale to many restaurants and handle periodic menu changes, this process needs to be dramatically faster.

Current Pain Points

  • 53 wines × 11 fields = ~580 manual data entries for a single restaurant
  • Basic info (producer, varietal, price) is available on menus, but sensory data is not
  • Restaurants update wine lists periodically, requiring re-entry
  • No bulk upload — wines are added one at a time via API
  • Sensory profile requires wine expertise to fill accurately

1 Photo/PDF Menu Upload with AI Parsing

Snap a photo of a restaurant's wine menu (or upload a PDF). Claude's vision API processes the image directly — no separate OCR step needed — and parses it into structured wine data with AI-estimated sensory profiles.

Workflow

  1. User takes a photo of the wine list or uploads a PDF/image
  2. Claude's vision API processes the image directly
  3. Claude parses the menu into structured JSON with sensory estimates based on wine knowledge
  4. AI returns structured JSON for all wines
  5. User reviews/edits in a confirmation UI, then submits

What You'd Build

  • New backend endpoint: POST /api/wines/restaurant/:id/parse-menu
  • Image upload handling (multer or similar)
  • Claude API integration for parsing + sensory estimation
  • Simple review/edit UI before final submission

Pros

  • Dramatically faster: photo → structured data in seconds
  • Claude has extensive wine knowledge for sensory estimation
  • Handles messy real-world menu formats
  • Works with any restaurant — just photograph their menu
  • No separate OCR step needed
  • Easiest for non-technical restaurant partners
  • Handles multi-language menus (Italian, French)

Cons

  • Claude API costs (~$0.01-0.05 per menu)
  • Sensory estimates are approximations
  • Poor photo quality reduces accuracy
  • Needs a review step — can't be 100% hands-off
  • Multi-page menus need multiple photos
  • Creative menu formatting can be hard to parse
Dev Effort: Medium (2-3 days) Ops Effort: Low

2 AI-Powered Sensory Auto-Fill (Smart Defaults)

Only require the "easy" fields (producer, varietal, region, type, price) and let AI automatically fill in the sensory profile. The user still enters basic info, but the tedious wine-knowledge fields are auto-generated.

Workflow

  1. User enters or pastes basic wine info (producer, varietal, region, type, price)
  2. Backend sends wine details to Claude API for sensory estimation
  3. AI returns: acidity, tannins, body, sweetness, and flavor profile
  4. Fields auto-populated — user can adjust if needed

What You'd Build

  • New backend endpoint: POST /api/wines/auto-fill-sensory
  • Claude API call with wine details → sensory profile
  • Frontend: "Auto-fill" button next to sensory fields
  • Batch mode support for CSV imports

Pros

  • Cuts data entry in half (5-6 fields instead of 11)
  • High accuracy — Claude knows wine characteristics by varietal and region
  • Can be applied retroactively to improve existing data
  • Simple to implement — one API call (or batch many)
  • User retains control to override
  • Cheapest option — minimal API cost

Cons

  • Still requires manual entry of basic fields
  • API cost per wine (~$0.005-0.01 each)
  • Blends or obscure wines may get less accurate estimates
  • Doesn't solve "getting basic menu data" problem
  • House/private-label wines harder to estimate
Dev Effort: Low (1 day) Ops Effort: Medium

3 Wine Database API Integration

Connect to existing wine databases (Vivino, Wine-Searcher, CellarTracker) that already have detailed wine information. Given a producer and varietal, auto-populate region, sensory profile, and other details.

Potential Data Sources

  • Vivino API — Largest consumer wine database with ratings and flavor profiles
  • Wine-Searcher API — Professional wine pricing/availability database
  • CellarTracker — Community-sourced tasting notes (40M+ reviews)
  • Open Wine Data — Open-source wine datasets on GitHub/Kaggle

Workflow

  1. User enters producer + varietal (or scans a barcode)
  2. Backend queries wine database API
  3. Auto-populates: region, type, sensory characteristics, typical price range
  4. User confirms and adjusts restaurant-specific price

Pros

  • Data from real tasting notes and expert reviews
  • Barcode scanning possible for bottled wines
  • Could auto-suggest wines as user types
  • Some databases include food pairing data
  • One-time integration, works for all restaurants

Cons

  • Most wine APIs are expensive or restricted (Vivino: no public API; Wine-Searcher: ~$500+/yr)
  • Restaurant wines differ from retail (house pours, exclusives)
  • Data format mismatch with SipSync's schema
  • Dependency on third-party service availability
  • Many Quartino wines (Antica Osteria, Rapido, Convivale) are restaurant-exclusive — unlikely to be in databases
Dev Effort: Medium-High (3-5 days + API costs) Ops Effort: Low (but gaps need manual fill)

4 Restaurant Admin Portal with Smart CSV Upload

Build a web-based admin interface where restaurant partners (or you) can upload a CSV with just the basic fields. The system handles parsing, validation, AI auto-fill, preview, and batch submission in one smooth flow.

Workflow

  1. Restaurant exports wine list from POS or types it into a spreadsheet
  2. Upload CSV with just: producer, varietal, type, price (minimum fields)
  3. System parses CSV, validates data, shows preview table
  4. "Auto-fill missing fields" button calls AI to populate sensory data, region, year
  5. Restaurant reviews and edits in an interactive table
  6. One click to publish all wines

What You'd Build

  • Admin page at /admin/upload (password-protected)
  • CSV parser (Papa Parse library)
  • Editable data table with inline editing
  • Bulk AI auto-fill integration (Idea 2 built in)
  • Batch API submission (all wines at once)
  • Diff view for menu updates ("3 wines added, 2 removed, 1 price changed")

Pros

  • Complete end-to-end solution for onboarding
  • Restaurant partners could self-serve
  • CSV upload + AI auto-fill = best of both worlds
  • Diff view makes menu updates easy
  • Validation catches errors before submission
  • Professional look for partner restaurants

Cons

  • Most significant development effort
  • Restaurants still need to provide basic wine data
  • Need to handle CSV format variations
  • Admin UI needs testing and maintenance
  • Security considerations for admin access
  • Restaurants may not want to do the work themselves
Dev Effort: High (4-7 days) Ops Effort: Very Low (self-serve)

5 Web Scraping Restaurant Wine Menus

Many restaurants publish wine menus on their website, Yelp, or OpenTable. A web scraper could pull this data automatically and parse it into the SipSync format with AI-assisted sensory estimation.

Potential Sources to Scrape

  • Restaurant's own website (wine/drinks menu page)
  • Yelp menu pages
  • OpenTable menu listings
  • Google Maps menu data
  • Toast/Square online ordering menus

Workflow

  1. Enter restaurant URL or search by name
  2. Scraper finds and extracts wine menu data
  3. AI parser structures into SipSync format (with sensory auto-fill)
  4. User reviews and submits

Pros

  • Fully automated for restaurants with online menus
  • No restaurant cooperation needed
  • Can be scheduled to catch menu changes
  • Free — no API subscription costs
  • Could build a library of pre-scraped restaurants

Cons

  • Most wine menus are PDFs or images, not structured HTML
  • Every website has a different layout — scrapers break constantly
  • Legal/ethical gray area (ToS, robots.txt)
  • Many restaurants don't have wine lists online
  • Third-party platforms often incomplete/outdated
  • Maintenance burden: scrapers break when websites change
  • Quartino's menu is a PDF — still needs OCR + AI (overlaps with Idea 1)
Dev Effort: High (5+ days + ongoing) Ops Effort: Medium (monitoring needed)

6 Hybrid "Quick Input" Mode — Paste & Parse

Instead of filling out a CSV with 11 columns, paste a raw wine list in any format (copied from a website, typed from a menu, dictated) and let AI parse it into fully structured wine records with sensory profiles.

Workflow

  1. User copies wine list text from anywhere (website, email, typed from menu, dictated)
  2. Pastes into a text box in the app
  3. Example input:
Whites:
Anterra Pinot Grigio Sicily $15
Heinz Eifel Riesling Mosel Germany $16.50
La Scolca Gavi Piedmont $18

Reds:
Avanti Chianti Tuscany $15
Sicilian Gold Nero d'Avola $16.50
Cune Crianza Rioja Spain $18
  1. Clicks "Parse" — AI structures the messy text into proper wine records with full sensory profiles
  2. User reviews in table format, makes corrections, submits

What You'd Build

  • Simple text area input + "Parse with AI" button
  • Claude API call to parse unstructured text → structured JSON with sensory data
  • Review/edit table
  • Batch submission to existing wine API

Pros

  • Lowest friction possible — paste any format, AI figures it out
  • Handles abbreviations and inconsistent formatting
  • Works with text from any source
  • AI fills in everything: full varietal names, regions, sensory data
  • Can handle "Caymus Cab" → "Caymus, Cabernet Sauvignon, Napa Valley"
  • Very fast to implement
  • Combines well with phone dictation

Cons

  • Accuracy depends on input quality
  • AI may misinterpret ambiguous entries
  • Still requires a review step
  • Doesn't automate initial data gathering
  • Claude API cost (~$0.02-0.05 per list)
  • Less structured than CSV for corrections at scale
Dev Effort: Low-Medium (1-2 days) Ops Effort: Low

Recommended Phased Strategy

Phase 1 — Quick Win

Implement Idea 2 + Idea 6 together. Build a "Paste & Parse" text box that sends raw wine list text to Claude API, which returns fully structured data including sensory profiles. This alone would have turned the Quartino task from hours to minutes.

Phase 2 — In-Person Scaling

Add Idea 1 (Photo/PDF + AI) for onboarding restaurants in person — just photograph their menu and go.

Phase 3 — Partner Self-Serve

Build Idea 4 (Admin Portal) when you have enough restaurants that your personal involvement doesn't scale. Bake in Ideas 1, 2, and 6 as input methods within the portal.
